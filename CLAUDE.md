# CLAUDE.md — AI 동반자 비서 (절대 규칙)

이 파일은 모든 작업 전에 읽는다. 여기 규칙은 **어기면 프로젝트가 무너지는** 것들이다.
세부 배경은 `docs/기획서.md`, 구조는 `docs/ARCHITECTURE.md` 참조.

---

## 0. 이 프로젝트가 무엇인가
감정(VAD)을 읽고 기억을 쌓으며 대화하는 **로컬 우선 데스크톱 AI 동반자**.
컴퓨터 조종과 발표는 **본체 위에 얹는 확장**이지 본질이 아니다.

스택: Electron + Next.js(renderer) + Python(sidecar) + 로컬 SQLite(WAL).

---

## 1. 절대 깨면 안 되는 규칙 (위반 = 아키텍처 붕괴)

### R1. 입력은 무조건 Message 객체로
모든 입력(채팅·음성·발표·시스템)은 본체에 들어가기 전 아래 형태로 통일한다.
```ts
type MessageSource = 'chat' | 'voice' | 'presentation' | 'system'
interface Message { source: MessageSource; text: string; timestamp: number }
```
본체는 source를 보고 동작을 분기하지 않는다. 어떤 source든 동일하게 처리한다.

### R2. 출력은 단일 이벤트 스트림 + 구독
LLM 출력은 하나의 스트림으로 흘리고, 소비자(자막·표정·TTS)가 각자 구독한다.
출력을 소비자별로 따로 생성하지 않는다.

### R3. 본체는 "발표"를 모른다
본체(대화 엔진) 코드에 `presentation`, `slide` 같은 발표 개념이 등장하면 **오염**이다.
발표는 본체를 호출만 하는 별도 컨트롤러(`renderer/src/presentation/`)에서 처리한다.
본체와 발표의 유일한 접점 = Message 객체 + 출력 스트림.
**검증법: 발표 폴더를 통째로 지워도 본체가 컴파일·동작해야 한다.**

### R4. 위험 도구는 게이트를 거친다 (코드로 강제)
모든 능력은 단일 도구 레지스트리에 등록하고 `risk` 태그를 단다.
- `safe` → 자동 실행
- `caution` → 실행 + 로그
- `dangerous` → **실행 직전 승인 게이트 필수** (delete, run_command, computer_use 등)

게이트는 도구 실행을 가로채는 코드다. **LLM 판단에 의존하지 않는다.**
모든 도구 호출은 감사 로그에 남기고, 가능하면 롤백 정보를 기록한다.

### R5. 감정은 작업 실행 결정에 개입하지 않는다
감정/유대(affection)가 바꿀 수 있는 것: 어투, 우선순위 **제안**, 소통 방식.
바꿀 수 없는 것: 작업의 **실행 여부**. risk 태그는 감정과 무관한 코드 상수다.
affection이 아무리 높아도 dangerous 게이트는 사라지지 않는다.

### R6. 기억은 로컬에만 둔다
대화·기억·감정 상태는 로컬 SQLite에 저장한다. 클라우드로 보내지 않는다.

### R7. API 키를 코드/번들에 박지 않는다
키는 사용자가 런타임에 입력한다. 난독화·암호화 동봉도 금지(추출 가능하므로).

## 2. 감정 시스템 핵심 규칙

- **VAD 3축**(Valence/Arousal/Dominance)을 LLM이 직접 추론한다. 룰/하드코딩 금지.
- **감정 추론과 답변 생성은 같은 LLM 호출**에서 한다. 별도 호출 금지(딜레이 0 유지).
- 출력 형식: 앞에 짧은 마커 → 뒤에 스트리밍 텍스트. **JSON 통짜 금지**(스트리밍 깨짐).
  - 예: `<vad>-0.6,0.7,0.3</vad> 무슨 일 있었어?`
  - 마커를 답변보다 **먼저** 출력(감정이 어투에 반영되도록).
- **스무딩**: 새 VAD는 이전 상태와 가중평균. 즉시 반영 금지.
- **감쇠(decay)**: 대화 없는 동안 VAD를 중립(0,0,0)으로 시간에 따라 되돌림.
- **표현(2D 고정)**: V·A → 표정/이모지, D → 어투 강도. 3D 아바타 금지.

## 3. 코드 작성 기준 (senior-dev)

- 한 함수 = 한 역할(20줄 초과 시 분리 검토). 한 파일 = 한 역할.
- 중첩 조건 3단계 이상 금지 → early return.
- 네이밍: 무엇인지 드러나게. 약어 금지(`msg`✕→`message`). Boolean은 is/has/can.
- TypeScript: `any` 금지. 반환 타입 명시. 유니온으로 상태 구분. null 체크 철저.
- 에러: 삼키지 말 것. 사용자에게 의미 있는 메시지 + `console.error('[위치]:', error)`.
- 매직 넘버/스트링 금지 → 상수. (스무딩 계수, 감쇠 계수, risk 임계 등)
- 엣지 케이스 항상 처리: 빈 값, null, **기억 0(첫 실행)**, 네트워크 실패, 사이드카 미준비.

## 4. 작업 순서 규칙

- **Core 완성 전 +1/+2에 손대지 않는다.** (범위는 ARCHITECTURE.md)
- 새 기능 전 자문: "이건 본체 수정인가, 확장점에 끼우는 것인가?" 본체 수정이면 R1~R5 재확인.
- import 경계(ARCHITECTURE.md §경계)를 어기는 import를 만들지 않는다.
- stream.ts는 emotion/vad에서 VAD 타입만 import함(런타임 의존 없음). 이후 stream.ts를 수정할 때 emotion 로직을 실행하는 import가 추가되면 경계 위반.
- 커밋 전: `npm run build` / `tsc --noEmit` / `lint` 통과 확인.

## 5. 알려진 함정 (검증된 것 — 반복 금지)

- **패키징은 초기에 1회 성공시켜 둔다.** (특히 Windows 빌드는 여러 번 실패할 수 있음. 마감 직전 첫 빌드 금지.)
- **Python 사이드카**: 자식 프로세스를 부모 세션에서 분리(좀비 방지), 출력은 로그 파일로 리다이렉트(버퍼 행 방지), 소켓/포트 바인딩 확인 후 호출(경쟁 조건 방지).
- **끼어들기**: 푸시투토크로 중단 시 재생·TTS생성·LLM생성·상태를 **함께** 취소(하나라도 빠지면 겹쳐 말함).
- **STT(음성 입력)는 Electron에서 WebSpeech로 동작하지 않는다(검증됨).** `SpeechRecognition`은 객체는 생성되나 구글 음성 백엔드·키가 동봉되지 않아 `start()`가 `network` 오류로 실패한다(Chrome에선 되지만 Electron 임베더엔 키 없음). `voice/stt.ts`에 녹음→변환 폴백 슬롯(`transcribeRecording`)이 설계돼 있으나 미배선 — 현재 음성 입력 비동작(TTS는 정상). 다시 켜려면 로컬 Whisper(사이드카) 또는 외부 STT를 변환기로 주입하되 키는 R7대로 런타임 입력. WebSpeech에 구글 키를 박는 우회는 금지(R7 위반).
- **preload 보안 표면**: 현재 sandbox:false 상태. preload는 얇은 통로로만 유지하고 도구 실행·민감 IPC 로직을 preload에 직접 넣지 말 것(메인 프로세스에 둘 것). sandbox 재활성화가 필요해지면 채널 정의 공유를 require 대신 빌드타임 주입으로 전환.
- **불확실하면 추측하지 말고 명시**한다. 같은 해결을 2회 이상 반복하지 않는다(다른 각도로).

## 6. 검증 규칙 — 모든 구현·수정 작업에 예외 없이 적용

모든 모듈 구현·수정·리팩토링 후, 아래 4단계를 반드시 수행한다.
이 4단계를 거치지 않고 "완료"로 보고하는 것을 금지한다.
작업이 작아 보여도 생략하지 않는다(생략 여부를 모델이 판단하지 않는다).

### ① 정적 검증
- `tsc --noEmit` (electron + renderer 양쪽) 에러 0
- `eslint` 경고 0
- import 경계(ARCHITECTURE §2) 위반 0 — 특히 core가 presentation/voice/expression/tools를 import하지 않는지 확인

### ② 서브에이전트 적대 검토 (필수)
- 여러 서브에이전트를 띄워 서로 다른 관점에서 결함을 탐색한다:
  - 경계/규칙 위반 관점 (R1~R7, import 경계)
  - 엣지 케이스 관점 (null·빈 값·첫 실행·동시성·중단/취소)
  - 경쟁 조건/비동기 관점 (레이스·좀비 프로세스·중복 호출)
  - "함수는 있는데 호출자가 없는" 고아 코드 관점
- 각 발견을 확정/오탐으로 판정하고, 확정된 것만 수정한다.
- 오탐은 기각 사유를 명시한다(틀린 지적을 반영해 코드를 악화시키지 않는다).

### ③ 런타임 검증
- 순수 로직은 단위 테스트로 핵심 케이스 실행 확인
- 가능하면 dev에서 실제 동작 라이브 확인
- 테스트 산출물은 확인 후 정리(레포에 잔재 남기지 않음)

### ④ 보고
- 위 ①~③ 결과를 보고에 포함한다:
  - 정적 검증 통과 여부
  - 적대 검토 발견 건수 / 확정·기각 내역
  - 런타임 검증 케이스와 결과
- "완료"만 적고 검증 내역을 생략하지 않는다.

### 같은 해결 반복 금지
- 동일한 수정·해결을 2회 이상 같은 방식으로 시도하지 않는다. 실패 시 systematic-debug 관점으로 다른 각도에서 접근한다.
