# AI 동반자 비서 — 기획서 (v1.0)

> 감정(VAD)을 읽고 기억을 쌓으며 대화하는 **로컬 우선 데스크톱 AI 동반자**.
> 컴퓨터 조종과 자기소개 발표는 본체 위에 얹는 확장 모듈이다.

작성 전제: **Core + 확장(+1) + 스트레치(+2)를 모두 구현한다는 가정** 아래 아키텍처를 설계한다.
구현은 단계적이되, 설계는 풀스코프. 나중에 확장 기능을 붙일 때 본체를 뜯어고치지 않도록 확장점을 미리 정의한다.

---

## 0. 한 줄 정의와 설계 철학

**정체성**: "감정을 읽고, 기억하고, 함께 사는" 데스크톱 AI 동반자. 컴퓨터 조종은 동반자가 가진 능력 중 하나이지 본질이 아니다.

**핵심 설계 철학 (전 문서를 관통하는 원칙)**
- **본체는 작게, 확장점은 미리.** 모든 확장 기능은 본체를 수정하는 게 아니라 정해진 확장점에 끼운다.
- **감정·유대는 톤에만, 작업 판단엔 개입하지 않는다.** 단 "판단·제안"까지는 하되 실행 결정권은 사용자에게 있다.
- **안전은 LLM 판단이 아니라 아키텍처로 강제한다.** 위험 통제를 코드 레벨에 박는다.
- **지연은 없앨 수 없으니 흐르게(스트리밍) 만들고 연출로 가린다.**
- **방지 가능한 오류·시간낭비를 구조로 차단한다.** 검증 가능한 것은 전부 미리 검증한다.

## 1. 범위 (Scope) — 3단 우선순위

설계는 전체를 포괄하되, 구현은 아래 순서로 진행한다. **Core가 완성되지 않은 상태에서 상위 단계에 손대지 않는다.**

### Core (반드시 완성 — 이것만으로 완결된 작품)
- 채팅(텍스트) 대화
- 단일 LLM 기반 의도 판단
- VAD 3축 감정 추론 (답변 생성과 **동일 호출**에서 처리)
- 감정 스무딩 + 시간 감쇠(decay)
- 2D 표정 캐릭터 (표정 + 눈 깜빡임)
- 2단 메모리 (로컬 SQLite, 단기 캐시 + 장기 영속, 주기 스냅샷, 사실 분리)
- 첫 실행 온보딩 (기억 0 상태 처리)

→ 결과물: **"감정을 가지고 기억하는 대화 동반자"**. 컴퓨터 조종·음성·발표가 전혀 없어도 작품으로 성립한다.

### +1 (시간이 되면)
- 음성 입출력 (스트리밍 STT / 스트리밍 TTS)
- 저지연 대화 파이프라인 (풀 스트리밍)
- 경량 모델 라우팅 (Core에선 단일 모델, 여기서 분기 도입)
- Layer 1 도구 몇 종 (파일·앱·검색)
- affection(유대도) 시스템

### +2 (스트레치 / 데모용)
- 발표 모듈 (PPTX 추출 + 슬라이드 렌더 + 실시간 발표 + 푸시투토크)
- Layer 2 화면 조작 (computer use, 통합 도구 레지스트리의 위험 도구로 흡수)

### 명시적 제외 (Non-Goals)
- **3D 아바타**: 2D로 고정한다. 3D는 추구하지 않으며, 표현 레이어를 3D 대비로 추상화하지도 않는다(over-engineering 방지).
- **친구 ID 공유 / 소셜 기능**: 동반자 본질과 무관하므로 제외한다.
- **완전 음성 barge-in(자동 끼어들기 감지)**: 에코/오감지 난이도가 높아 추구하지 않는다. 끼어들기는 푸시투토크로 한정한다.
- **클라우드 기억 저장**: 기억은 로컬에 둔다. 프라이버시·오프라인·컨셉 일관성 때문.

## 2. 아키텍처 원칙 (확장 호환성을 위한 5대 선제 설계)

각 원칙은 "확장 시 어디서 충돌이 터지는가"를 미리 막기 위한 것이다.

### 원칙 1 — 입력 추상화 (채팅과 음성을 하나로)
모든 입력을 처음부터 **통일된 메시지 객체**로 만든다.

```
Message {
  source: "chat" | "voice" | "presentation" | "system"
  text: string
  timestamp: number
}
```

- Core의 채팅도, +1의 STT 결과도, +2 발표의 멘트 요청도 전부 같은 `Message`로 본체에 들어간다.
- **효과**: 음성(+1)을 붙일 때 "STT가 같은 Message를 만들어 주입"하면 끝. 본체 파이프라인은 변경 0.

### 원칙 2 — 출력은 단일 이벤트 스트림 (구독 모델)
LLM이 토큰을 흘리면, 그 출력을 **여러 소비자가 각자 구독**한다.

```
출력 스트림 ──┬── 자막/채팅 렌더러 (Core)
              ├── 2D 표정 컨트롤러 (Core)
              ├── TTS 재생기 (+1)
              └── (필요 시 추가 구독자)
```

- 한 소스, 여러 소비자. 출력을 따로따로 만들지 않는다.
- **효과**: +1 TTS도 "스트림 구독자 추가"로 끝난다. 표정·자막·음성 동기화가 한 소스에서 보장된다.

### 원칙 3 — 통합 도구 레지스트리 + 위험도 게이트
모든 능력(파일/앱/검색/화면조작)을 **하나의 도구 레지스트리**에 등록하고, 각 도구에 **위험도(risk level)**를 태그한다.

| 도구 예시 | risk | 동작 |
|---|---|---|
| read_file, search_web | safe | 자동 실행 |
| write_file | caution | 실행 + 로그 |
| delete_file, run_command, **computer_use** | dangerous | **승인 게이트 통과 필수** |

- LLM 입장에선 전부 동일한 "도구"다. Layer 1/2 구분을 LLM이 몰라도 된다. Layer 2(화면조작)는 "위험도 높은 도구 하나(computer_use)"로 흡수된다.
- **승인 방식 = 에스컬레이션 트리거**: 평소 자동 실행, 위험(비가역·민감) 작업에서만 멈춰 승인 요청. (모든 작업 승인은 피로·습관적 승인 리스크를 유발하므로 채택하지 않음.)
- **게이트는 도구 실행 직전 코드가 가로챈다.** LLM의 선의에 의존하지 않는다.
- **감정과의 관계**: 위험도 태그는 코드 상수다. affection이 아무리 높아도 dangerous 도구의 게이트는 사라지지 않는다 → "감정은 작업 판단에 개입 안 함" 안전선이 아키텍처로 강제된다.
- 모든 도구 호출은 **감사 로그**에 남기고, 가능한 작업은 **롤백** 정보를 함께 기록한다.

### 원칙 4 — 발표는 본체를 모르는 격리 컨트롤러
본체(대화 엔진)는 발표의 존재를 **모른다**. 발표 모드는 본체 바깥에서 본체를 "운전"하는 별도 레이어다.

```
[본체] 입력 Message → 감정 추론 → 답변 생성 → 출력 스트림   (발표 코드 0줄)

[발표 컨트롤러] (본체를 호출만 함)
  상태: 현재 슬라이드, 슬라이드별 내용/노트, 진행 단계
  - 슬라이드 내용을 Message(source:"presentation")로 본체에 주입 → 본체가 평소처럼 답변(=발표 멘트)
  - 푸시투토크 질문도 Message로 주입 → 본체가 평소처럼 답변
  - 본체 출력을 받아 TTS + 슬라이드 넘김
```

- 본체와 발표의 **유일한 접점 = Message 객체 + 출력 스트림**. 그 외 연결 금지.
- **격리 규칙 (반드시 준수)**:
  - 본체 코드는 "발표", "슬라이드" 같은 개념을 알면 안 된다. 알면 오염된 것.
  - 발표 컨트롤러는 본체를 호출만 한다. 본체 내부를 건드리지 않는다.
- **효과**: 발표 모듈을 통째로 들어내도 본체는 1줄도 바뀌지 않는다. 발표 중에도 감정·기억이 그대로 작동한다(발표가 본체를 거치므로). "자기 아키텍처 설명" 퍼포먼스도 발표 컨텍스트 주입만으로 자연스럽게 구현된다.

### 원칙 5 — Python 사이드카 골격을 Core부터 깔아둔다
Core는 Python이 거의 불필요하지만, +1(STT)·+2(PPTX)는 Python이 필요하다. 나중에 급조하면 좀비 프로세스·버퍼 행 함정을 마감 직전에 만난다.

- **Core 단계에서 사이드카 인프라(생명주기 관리 + 통신 채널)만 미리 구축**한다. 기능은 비어 있어도 됨.
- 검증된 함정 선제 차단:
  - 자식 프로세스를 부모 세션에서 분리 → 앱 종료 후 좀비 프로세스 방지
  - 사이드카 출력을 전용 로그 파일로 리다이렉트 → 버퍼 관련 행(hang) 방지
  - 소켓/포트 바인딩 체크로 "준비됨"을 확인한 뒤 호출 → 경쟁 조건 방지
- Python 패키징은 단일 실행파일(PyInstaller 류)로 freeze 후 사이드카로 번들.

## 3. 감정 시스템 (VAD)

### 3.1 모델
- **VAD 3축**: Valence(긍·부정), Arousal(각성·이완), Dominance(주도·위축). 심리학 표준 차원 모델.
- 룰/하드코딩이 아니다. LLM이 사용자 발화의 말투·단어·맥락을 읽고 세 값을 직접 추론한다.

### 3.2 추론 방식 (딜레이 0 설계)
감정 추론을 **답변 생성과 동일한 LLM 호출**에서 처리한다. 별도 호출 금지.

- 출력 형식: **앞쪽에 짧은 감정 마커 + 뒤쪽에 자유 텍스트(스트리밍)**.
  - 예: `<vad>-0.6,0.7,0.3</vad> 무슨 일 있었어? 목소리가 좀...`
  - JSON 통짜 금지 (완결돼야 파싱되므로 스트리밍이 깨진다).
- **마커를 답변보다 먼저** 출력시킨다 → 감정이 먼저 결정되어 그 감정에 맞는 어투로 답변이 생성된다.
- 앞쪽 마커만 파싱하고, 그 뒤 토큰은 스트리밍으로 TTS/렌더러에 즉시 흘린다 → 구조화 출력과 스트리밍 양립.

### 3.3 안정화 — 스무딩 + 감쇠
- **스무딩**: LLM의 VAD 출력은 호출마다 흔들린다. 새 값을 즉시 반영하지 않고 이전 상태와 가중평균하여 갱신한다. (감정이 갑자기 튀지 않고 자연스럽게 변하는 효과도 겸함.)
- **감쇠(decay)**: 대화가 없는 동안 VAD를 시간 경과에 따라 중립(0,0,0)으로 서서히 끌어당긴다. (사용자가 화나게 한 뒤 침묵해도 동반자가 영원히 화나 있지 않도록.) 구현은 경과 시간 × 감쇠 계수.

### 3.4 표현 (2D, 표현 채널 분리)
"3D 감정 시스템"을 표방하므로 D축을 버리지 않는다. 단 이모지의 표현 한계 때문에 **채널을 분리**한다.

- **V·A → 2D 캐릭터 표정 / 이모지**: 표정 격자 매핑. 경계 전환은 스무딩·히스테리시스로 부드럽게.
- **D → 어투 강도/자신감**: 높은 D는 단정적 어투, 낮은 D는 위축된 어투. 말투로 발산.
- 눈 깜빡임은 타이머 기반 프레임 전환.
- → "이모지엔 V·A를, 어투엔 D를 매핑해 표현 채널을 분리했다"가 일관된 설명이 된다. 감정 엔진은 진짜 3D로 돈다.

### 3.5 한계 (명시)
- 텍스트 기반 감정 추정은 빈정거림·농담을 자주 틀린다.
- LLM VAD 추정은 일관성이 흔들린다(스무딩으로 보정).

## 4. 주체성 경계

최초 비전("주체적 판단과 감정을 바탕으로 한 작업")과 안전선("작업 판단 개입 금지")을 화해시키는 경계:

- 감정/상태가 바꿀 수 있는 것: 작업의 **방식, 우선순위 제안, 소통 톤**.
- 감정/상태가 바꿀 수 없는 것: 작업의 **실행 여부 결정**.
- 예: 사용자가 지쳐 보이면(낮은 V·낮은 A) "지금 이건 무리하지 말고 핵심만 할까요?"라고 **제안**한다. 실행 결정권은 사용자.
- 한 줄: **"판단·제안까지는 하되, 실행 결정은 사용자가 한다."**

## 5. 메모리 시스템

### 5.1 구조 (메모리 계층)
- **단기기억 (세션 캐시, 휘발성)**: 앱 켠 순간부터 끌 때까지 대화 맥락 유지.
- **장기기억 (영속 저장)**: 종료 시 단기기억을 요약·정리해 넘긴다. 다음 실행 때 요약을 먼저 읽어 맥락을 파악한다.

### 5.2 견고성 보강
- **주기적 중간 스냅샷**: 종료가 항상 정상적이지 않다(크래시·강제종료). N턴마다 또는 일정 시간마다 단기기억을 영속 저장해 통째 증발을 방지한다.
- **사실 분리 추출**: 요약은 손실 압축이다. 중요한 사실(선호·약속·이름)은 요약과 별개로 **구조화 키-값**으로 따로 저장한다.
- **단일 쓰기 경로**: 세션 중 쓰기와 종료 시 쓰기를 직렬화해 동시 쓰기 충돌을 방지한다.

### 5.3 첫 실행 (기억 0 상태)
- 빈 요약을 읽었을 때의 기본 상태를 정의: 중립 감정(0,0,0), affection 0, 자기소개 첫인사(온보딩).
- 첫 실행이 어색하거나 깨지지 않도록 기본 동작을 명문화한다.

### 5.4 스키마 설계 (확장 대비)
- 저장소: **로컬 SQLite** (WAL 모드).
- 테이블 분리: 기억 / 감정 상태 / 관계 지표(affection 등) / 메타.
- affection(+1) 같은 확장 필드는 미리 자리를 잡아두거나 유연한 보조 구조(키-값 테이블)로 둔다 → 확장 시 스키마 마이그레이션 지옥 방지.
- (친구공유 등 소셜 스키마는 범위에서 제외하므로 고려하지 않는다.)

## 6. 컴퓨터 조종 (도구 시스템)

- **Layer 1 (정의된 도구)**: 파일·앱·검색 등. 빠르고 안전·안정적. +1에서 도입.
- **Layer 2 (화면 조작, computer use)**: 정의되지 않은 작업의 fallback. 느리고 불안정(베타). +2에서, 통합 도구 레지스트리의 `computer_use`(dangerous)로 흡수.
- 모든 도구는 **원칙 3(통합 레지스트리 + 위험도 게이트 + 감사 로그 + 롤백)** 을 따른다.
- 화면조작은 데모 한두 케이스로 한정한다. 메인 동작은 Layer 1으로 처리한다.

## 7. 저지연 대화 파이프라인 (+1)

지연은 물리적으로 0이 될 수 없다(클라우드 API의 하한). "없애기"가 아니라 "숨기기"가 목표.

- **풀 스트리밍**: STT·LLM·TTS 전 구간 스트리밍. "다 기다렸다 한 번에"를 제거.
- **단일 호출 합치기**: 감정 추론을 답변 생성과 한 호출로 묶어 추가 왕복 0 (§3.2).
- **저지연 스트리밍 TTS 선택**: 핵심 지표는 전체 합성 시간이 아니라 **첫 바이트까지의 시간(TTFB)**. 구현 시점에 실제 비교 측정해 선택.
- **연출로 공백 메우기**: 질문 즉시 표정을 "생각 중"으로 전환(반응 0초) → 물리 지연을 체감상 가린다. 비용 0, 효과 큼.
- **끼어들기 = 푸시투토크**: 키/버튼으로 끊으면 TTS를 **하드컷**으로 즉시 중단. 동시에 진행 중인 TTS 생성·LLM 생성·재생 버퍼를 **함께 취소**하고 상태를 리셋한다(4가지 동시 취소). 하나라도 빠지면 사용자 말에 겹쳐 말하거나 끊긴 뒤에도 마저 말한다.
- **컴포넌트별 지연 측정**: STT first-token, LLM first-token, TTS first-byte를 분리 계측해 병목을 추측이 아니라 데이터로 잡는다.
- **물리 하한 인지**: 해외 리전 API 사용 시 왕복이 수백 ms 추가될 수 있다. 가능한 빠른 회선을 확보한다(코드로 못 막는 변수는 대비만).

## 8. 발표 모듈 (+2)

- **목적**: AI가 자기가 만들어진 방법·작동 원리를 대화로 설명하는 퍼포먼스. 끼어드는 질문에 실시간 반응.
- **본체 격리**: 원칙 4를 따른다. 발표 컨트롤러가 본체를 운전하며, 본체는 발표를 모른다.
- **슬라이드 처리**:
  - 내용/발표자 노트 추출: python-pptx(또는 순수 파이썬 대안)로 슬라이드별 텍스트·노트 추출. (검증됨)
  - 슬라이드 렌더링: python-pptx로는 이미지화가 안 된다. **별도 변환(LibreOffice 헤드리스 등)으로 슬라이드를 이미지화해 앱 내부에서 직접 렌더링**한다. PowerPoint에 의존하지 않아 OS/버전/해상도 변수를 제거.
  - LibreOffice 변환이 특정 폰트·애니메이션을 깨뜨릴 수 있으므로 **발표 전 리허설로 변환 결과를 반드시 검증**한다.
- **발표 내용**: 실시간 LLM 생성(미리 생성 폐기 — 즉석 질문 대응 위해).
- **끼어들기**: 푸시투토크(§7).
- **메타 장치**: 발표 컨트롤러가 본체에 주입하는 컨텍스트에 이 앱의 아키텍처 요약을 포함 → 본체가 자기 구조를 실제로 알고 설명. 환각 방지를 위해 핵심 질문은 사전 테스트.
- **API 키**: 키를 클라이언트에 동봉/난독화하는 것은 안전하지 않다(추출 가능). **사용자(키 보유자)가 실행 시 자기 키를 입력**하는 방식.
- **남는 변수**: 네트워크(실시간이라 인터넷 필수 → 핫스팟 백업), 지연(연출로 가림).

## 10. 비용·운영 원칙

- **Core 단일 모델**: 라우팅은 +1로. (단일 호출 합치기와 라우팅 판단이 충돌하므로 Core에선 단순화.)

## 부록 A — 용어

- **VAD**: Valence(쾌-불쾌)·Arousal(각성-이완)·Dominance(주도-위축) 3차원 감정 모델.
- **스무딩(smoothing)**: 새 측정값을 이전 상태와 가중평균해 급변을 완화하는 기법.
- **감쇠(decay)**: 시간 경과에 따라 상태를 기준값(중립)으로 서서히 되돌리는 처리.
- **사이드카(sidecar)**: 메인 앱과 별도로 함께 실행되는 보조 프로세스(여기선 Python).
- **에스컬레이션 트리거(escalation trigger)**: 평소 자율 실행하되 위험 신호 감지 시에만 인간 승인을 요청하는 게이트 방식.
- **TTFB(time to first byte)**: 요청 후 첫 응답 바이트(여기선 첫 음성)까지의 시간. 실시간 음성의 핵심 지표.
- **푸시투토크(push-to-talk)**: 키/버튼을 눌러 발화 입력을 트리거하는 방식. 자동 음성 감지의 에코/오감지 문제를 회피.
